01 需求分析 Q1 / 2
Question 1
举例说明一个你喜欢的产品设计，并阐述这个产品设计解决了用户的什么需求。

微信读书「热门划线」

用户在微信读书中对文本划线后，系统聚合所有人的划线数据，将被多人标记的段落以虚线下划线 + 人数角标展示给后续读者。点击划线可查看他人对同一段落的评论。

解决的四个需求

需求用户面临的问题功能的解决方式
信息过滤 一本书几十万字，无法快速判断哪些段落值得精读。 热门划线本质是众包筛选——被千人划过的段落，大概率是全书最有价值的内容。读者可以扫一遍划线快速判断这本书是否值得投入时间。这个功能把阅读前的筛选成本从「翻几十页目录和书评」降到了「滑三屏划线」。
共读体验 阅读是单人行为，但人有交流的本能需求。 看到「有 3,241 人划线」时，读者知道其他人也在同一句话前停留过。不需要加入读书会、不需要发朋友圈，但获得了一种低摩擦的共读感。这种设计把社交行为降到了几乎为零的成本。
低门槛表达 普通读者有表达欲，但写书评的心理和精力成本太高。 划线不需要组织语言，手指一划就能参与内容生产。当自己的划线被他人点赞时，获得品味被认可的反馈。相比写书评，划线的内容生产成本趋近于零，但表达的满足感并不低。
文本级讨论 传统书评区只能讨论整本书，无法聚焦到具体句子。 点击划线进入的「想法」评论，讨论粒度精确到一句话。用户看到的是围绕某一段文字的观点碰撞，而不是泛泛地评价整本书。这种粒度在传统书评区无法实现。

一个交互动作（手指划过屏幕）同时覆盖了功能需求（信息过滤）和情感需求（表达、社交），且不增加认知负担。划线越多 → 后读者体验越好 → 读完率越高 → 更多人划线 → 数据更丰富。这个循环意味着产品的阅读体验随用户规模增长而改善，而不是像大多数产品那样随规模增长而稀释。

02 交互设计 · Agent时代的生成式界面 微信重设计
Question
选取一个你喜欢的产品，尝试用生成式界面来重新设计它。
选择产品：微信

微信当前界面的约束

微信的界面克制带来了一个后果：朋友圈、视频号、搜一搜、小程序、直播、购物、群收款、接龙等功能散落在不同入口，用户需要在多个页面之间手动跳转。功能数量与界面简洁之间的矛盾，在固定UI时代只能通过产品经理的取舍来解决——而取舍意味着总有一部分用户的需求被牺牲。

生成式界面改变的是这个前提：当界面可以按需生成和销毁时，不需要在「简洁」和「能力」之间取舍。界面的控制权从产品经理手中转移到用户当下的意图上。

三层界面体系

层级策略设计场景
静态骨架 永久保留 聊天列表、通讯录、「我」。
这三个 Tab 是微信作为通讯工具的基础设施。用户对它们有稳定的空间记忆，不应随上下文变化。 		任何时候打开微信，这三个 Tab 始终存在。
Agent 对话 自然语言驱动 聊天列表顶部增加一个 Agent 对话入口。用户用自然语言表达意图：
· 查找历史文件：「找上周三和老板讨论的PDF」
· 发起协作：「在群里发起周六聚餐投票」
· 内容发现：「最近有什么热门的小程序游戏」 		用户说「帮我订三张下周去北京的机票」，Agent 调用小程序比价、生成确认卡片，整个过程在对话流中完成。
动态页面 用完即销毁 Agent 根据上下文实时生成临时页面：群收款自动识别AA场景生成分摊卡片；旅行规划聚合机票/酒店/攻略为 Dashboard；活动报名自动生成含时间/地点/人数的表单。页面在任务完成后 24 小时自动销毁，关键信息归档到群聊。 8 个朋友讨论周末自驾游 → Agent 生成行程页：路线投票、住宿选择、费用分摊、天气提醒。周日结束后摘要归档，页面消失。

朋友圈的变化

  • 摘要模式：早上打开，Agent 输出一段文字——「你关注的 3 个人昨晚发了新动态，小明的旅行照片有 42 个赞」。用户决定是否展开查看。
  • 意图驱动的 Feed：对 Agent 说「最近有什么好玩的」，它从朋友圈、视频号、小程序中拉取内容生成一个临时推荐流，按当下状态排序，不按时间排序。
  • 社交搜索：「最近有谁在学 Python？」——Agent 扫描好友动态和收藏，列出相关的人和内容。纯时间线做不到这种查询。

静态层给空间锚点——用户始终知道自己在哪里；对话层给灵活性——自然语言替代菜单导航；动态层给能力——Agent 生成界面而非用户寻找界面。

新问题

  1. 可发现性丧失：功能完全由 Agent 触发后，用户可能永远不知道某些能力存在。需要在对话中做能力提示——「我还可以帮你……」
  2. 隐私边界：Agent 需要跨聊天、跨朋友圈扫描信息才能智能响应，与微信的隐私优先原则冲突。需要细粒度授权——用户明确选择哪些数据域对 Agent 可见。
  3. 错误成本高：传统界面点错可以返回。Agent 理解错意图时，整个生成页面都是错的。需要每一步都保留撤销入口。
  4. 表达负担：用户不再需要记忆功能位置，但需要学会描述需求。对不擅长表达的用户，需要引导式模板而非空白输入框。

05 统计学 Q1 + Q2 · 必答
Question 1
请用简洁的非数学语言描述以下概念，并举一个这个概念在产品工作中应用的例子。

1. 方差 (Standard Deviation)

方差衡量一组数据围绕均值的分散程度。两个数据集可以有相同的均值，但方差完全不同。均值回答「典型情况是什么」，方差回答「个体之间差多远」。

产品应用：某视频 App 的日均观看时长均值 35 分钟，但标准差高达 42 分钟。这说明用户严重两极分化：一部分几乎不打开，另一部分重度使用。只看均值会误以为用户活跃度尚可。团队根据方差信号将用户分为「待激活」和「深度用户」两个群体，分别设计运营策略，整体留存提升 12%。不看方差，这个分层机会不会被发现。

2. F1 Score

F1 是精确率（识别为正的样本中有多少确实为正）和召回率（实际为正的样本有多少被识别出来）的调和平均。算术平均会被极端值拉偏，调和平均对两者的平衡更敏感——只要有一个很低，F1 就会很低。

产品应用：内容审核系统识别违规评论。只追求高精确率（宁可漏掉，不可误删），社区氛围恶化；只追求高召回率（宁可误删，不可漏掉），用户投诉飙升。F1 同时约束两个方向：精确率 92%、召回率 85% 时 F1 = 88.4%。单独看精确率或召回率都会遗漏另一半信息，F1 给了一个统一的优化目标。

3. 中心极限定理 (加分项)

不管原始数据的分布是什么形状（偏态的、扁平的、多峰的），只要反复从总体中随机抽样，每次计算样本均值，这些样本均值的分布会随着样本量增大而趋近正态分布。

产品应用：A/B 测试只需要几万用户就能推断几千万用户的效果，底层依赖的就是中心极限定理。它保证了样本量足够时，样本均值的分布近似正态，可以用正态分布的性质计算置信区间和 p 值。即使用户付费金额严重右偏（少数大R贡献大部分收入），样本量 1,000+ 后仍然可以用 t 检验判断实验效果。没有这个定理，每次实验都需要全量用户参与。

4. 贝叶斯定理 (加分项)

贝叶斯定理描述当获得新信息时，如何更新对某事件的概率判断。它从先验概率出发，用新证据修正，得到后验概率。

产品应用：电商垃圾评论过滤。先验概率：全站评论有 2% 是垃圾。当一条评论包含「加微信」时——垃圾评论含该词的概率 40%，正常评论含该词的概率 0.1%——贝叶斯定理将这条评论是垃圾的概率从 2% 更新到约 89%。实际的垃圾过滤系统（朴素贝叶斯分类器）就是基于多个这类特征持续更新概率判断。关键在于系统能随着新数据的积累自我修正，而不是依赖一次性规则。

Question 2
对话产品只关注线上用户对话的整体可用率，会导致什么问题？

整体可用率掩盖四类问题

问题表现为什么整体指标看不到
1. 长尾场景崩溃 90% 用户问常见问题（查订单、退货、物流），Agent 处理得很好。10% 用户问复杂问题（跨境税费、组合优惠叠加、预售定金规则），Agent 完全无法处理，用户多次转人工失败后离开。 整体可用率 = 90% × 95% + 10% × 20% = 87.5%，看上去正常。但 10% 的复杂场景用户经历了极差体验，且他们通常是高价值用户——买得多，问题才复杂。
2. 弱势用户被系统性忽视 老年用户用方言或不规范表达，Agent 准确率远低于年轻用户。视障用户依赖屏幕阅读器，Agent 的富文本卡片反而造成障碍。 这些群体占比小（如老年用户 5%），即使他们的可用率只有 50%，对整体指标的影响仅 2.5 个百分点。只看整体，团队永远不会意识到自己在系统性忽视某一类用户。
3. 辛普森悖论 版本迭代后整体可用率从 85% 升至 87%，团队认为优化有效。实际情况：简单场景可用率从 95% 降到 93%（新模型在某些常见 query 上变差），复杂场景从 20% 升到 30%。整体上升完全是因为复杂场景用户占比下降——他们之前体验太差，已经弃用。 当子群体构成比例发生变化时，整体指标的趋势可能与每个子群体的趋势完全相反。只有拆开看分群指标才能发现真实情况。
4. 响应正常但未解决问题 Agent 每次都能正常回复，不报错、语法正确、语气友好——可用率很高。但用户真正想解决的问题（如「这个和那个哪个更适合我的肤质」），Agent 给出泛泛对比，用户还是得去小红书查攻略。 可用率通常只衡量系统是否正常响应、没有错误，不衡量响应是否解决了用户的问题。「上菜快」和「菜好吃」是两回事。

正确的监控方式

  1. 按场景分层：高频简单、中频中等、低频复杂场景分别监控可用率和满意度
  2. 按用户分层：新老用户、不同活跃度、不同年龄段，分别看指标
  3. 引入结果指标：用「任务解决率」替代「可用率」——通过会话结束后的满意度调研、人工抽样、重复求助率（同一问题 24h 内再次发起）来衡量
  4. 监控最差切片：主动找出表现最差的 10% 场景和 10% 用户群体，专项优化

整体可用率是一个「让坏消息消失」的指标。它让团队对高价值用户流失、弱势群体被忽视、迭代方向被误导等情况一无所知。好的监控体系不是追求一个好看的汇总数字，而是让每个子群体的问题都能被单独看到。

06 Agent设计 Q2 / 2
Question 2
对一个自媒体视频的最高赞赏，就是发朋友圈/朋友。同理，一个购物Agent，要设计什么样的功能，用户会很乐意发朋友圈/朋友呢？

分享的前提

用户转发购物 Agent 的内容，动机不是推荐工具本身，而是展示自己的某个侧面——精明、品味独特、发现了别人不知道的好东西。功能设计需要回答的问题是：Agent 帮用户产生了什么值得展示的「关于我」的素材。

四个可分享的功能

#功能用户展示什么设计朋友圈示例
1 省钱战绩卡片 精明 用户通过 Agent 比价/找券/凑满减后，自动生成战绩卡片：原价 vs 实付价对比、找到的优惠券数量、跨平台比价次数、一条总结文案。 「同一台洗衣机，Agent 帮我跨了 7 个平台比价，省了 800 块。以前买东西从来不做功课……」
[配图：省钱战绩卡片]
2 购物人格报告 独特 基于用户通过 Agent 的购物行为，生成季度/年度报告：购物人格标签（如「参数对比狂」「颜值优先型」）、年度最值的一笔消费、与全国用户的对比数据。 「我的购物人格：参数对比狂。Agent 说我每次买东西平均对比 5.7 个参数，比 98% 的人更纠结。但年度最值的一笔——对比了 17 款才下单的人体工学椅，确实好用。」
[配图：购物人格报告长图]
3 决策投票卡片 社交 用户在两三个商品间纠结时，Agent 生成可分享的投票卡片：关键参数对比、朋友可投票建议、投票后看结果。朋友投票后可获得自己的推荐。 「三款咖啡机纠结一周了，Agent 把参数都扒干净了还是选不出来。帮我投个票，选完请你喝第一杯。」
[配图：三款咖啡机参数对比投票卡片]
4 宝藏发现卡片 品味 Agent 推荐了一个小众高匹配商品时，生成「发现故事」卡片：为什么这个商品适合你（基于历史偏好分析）、在什么圈子里口碑好、「全平台仅 X% 用户知道这个品牌」。 「Agent 给我推了一个北欧小众香薰品牌，说和我之前买的 5 件东西风格高度一致。全淘宝只有 3% 的人知道这个牌子。它比我自己更清楚我审美。」
[配图：宝藏发现卡片]

触发分享的时机

1
情绪高点
省钱、发现宝藏、收到朋友投票结果时触发
2
零编辑
Agent 预生成图和文案，用户一键发送
3
自我表达
每张卡片回答「我是怎样的人」
4
社交钩子
朋友可以参与（投票），不是单向炫耀

用户分享的不是 Agent，是 Agent 帮他们生产的「关于我」的内容。当朋友看到省钱战绩或购物人格后问「这是什么 App」，新增用户来自社交链而非投放。衡量指标：分享率（目标 > 15%）、分享回流率（新用户中来自分享链接的比例）、各功能分享占比（避免依赖单一类型）。
